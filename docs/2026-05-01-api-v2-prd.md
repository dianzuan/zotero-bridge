# Zotron v2 PRD — 2026-05-01

CLI-first, agent-native redesign. The CLI is the primary interface for LLM agents; JSON-RPC is the substrate that CLI calls, not the interface agents interact with directly.

**Design inputs:**
- User feedback from 3 agent sessions (parameter guessing failures, namespace confusion, method-not-found loops)
- Internal audit (82 RPC methods, 36 divergence rows from previous PRD)
- Competitive benchmark against Zotero Web API v3, Notion API, Obsidian Local REST API, Logseq Plugin API, Linear API
- CLI vs RPC research: Anthropic data shows CLI (bash tool) is 35-98x more token-efficient than structured tool schemas; trained-in bash reliability ~100% vs custom tools ~72%

**Non-goals:**
- MCP tool exposure (agent-facing interface is CLI via bash tool, not MCP)
- Backward compatibility with v1 RPC method names (skill docs handle migration)
- Cursor-based pagination (offset is correct for single-writer local DB)

---

## 1. Design Principles

### 1.1 CLI-first

The agent's primary interface is the `zotron` CLI via Claude's trained-in bash tool (245 tokens fixed overhead). RPC is the transport layer, not the agent-facing API.

**Why:**
- Bash tool is trained-in: Anthropic optimized Claude on thousands of successful bash trajectories
- Schema cost: 82 RPC methods as MCP tools would cost ~8,000+ tokens per turn; CLI costs 245 tokens flat
- Discovery: `--help` is on-demand (zero cost until invoked); RPC schemas load every turn
- Reliability: CLI ~100% vs structured tools ~72% (Reinhard 2026, 75-test benchmark)
- Industry consensus: gh, kubectl, stripe cli, supabase cli all use this pattern

**Consequence:** Every RPC method must have a corresponding CLI subcommand. `zotron rpc` remains as an escape hatch for edge cases, not the primary path.

### 1.2 One guess, one hit

An agent wanting PDF text should guess `zotron items fulltext 12345` — and that should work. The subcommand name, flag names, and output format must all match the agent's first instinct.

**Test:** for any user intent, can an LLM produce the correct `zotron <noun> <verb> <args>` call without reading docs? If not, the CLI is wrong.

### 1.3 No multi-step dance

An operation that a human would describe as one step must be one CLI call. "Get the PDF text for this paper" should not require `zotron attachments list --parent 12345` → find PDF ID → `zotron attachments fulltext <pdf-id>`.

### 1.4 Consistent contracts

- Same parameter name = same semantics everywhere
- Same return envelope for all list operations
- Same confirmation shape for all mutations

### 1.4 Self-describing

The API must be discoverable at runtime via introspection methods. An agent that knows nothing about zotron should be able to call `system.describe` and learn every method, parameter, and return shape.

### 1.5 No redundancy

One canonical method per operation. No aliases, no convenience duplicates. The skill layer teaches agents the correct method names.

## 2. Identifier Strategy

### 2.1 Items: `id` accepts `number | string`

All item-scoped methods accept a single `id` parameter that resolves:
1. Numeric Zotero ID: `42`
2. 8-char alphanumeric key: `"YR5BUGHG"` (resolved via `Zotero.Items.getByLibraryAndKeyAsync`)
3. Numeric string: `"42"` (parsed to number, then resolved as numeric ID)

Implementation: `requireItem(idOrKey: number | string)` in `src/utils/guards.ts`.

### 2.2 Collections: `id` accepts `number | string`

All collection-scoped methods accept `id` that resolves:
1. Numeric Zotero ID: `79`
2. 8-char alphanumeric key: `"COL12345"` (resolved via `Zotero.Collections.getByLibraryAndKeyAsync`)
3. Collection name: `"产业经济学年会"` (resolved via name match on `Zotero.Collections.getByLibrary`)

Implementation: extend `requireCollection(idOrKey: number | string)` with same resolution logic as `requireItem`, plus name-based lookup.

### 2.3 Parameter naming

| Parameter | Meaning | Used in |
|-----------|---------|---------|
| `id` | The entity this method operates on | All `get`, `update`, `delete`, `trash`, `restore` methods |
| `parentId` | The parent entity (item or collection) | `notes.list`, `notes.create`, `attachments.list`, `annotations.list`, `collections.create` |
| `ids` | Batch of entity IDs | `batchTrash`, `mergeDuplicates`, export methods |

Never use `itemId`, `collectionId`, `collectionID`, or `relatedId`. When a method takes two items (e.g., add-related), use `id` + `targetId`.

## 3. Return Shape Contracts

### 3.1 List envelope

Every method returning multiple entities uses:

```json
{
  "items": [...],
  "total": 166,
  "limit": 25,
  "offset": 0,
  "hasMore": true
}
```

Fields:
- `items`: array of serialized entities (always present, may be empty)
- `total`: total count before pagination (database-side count, NOT slice length)
- `limit`: page size used (default 50 if caller omitted)
- `offset`: starting offset (default 0)
- `hasMore`: `total > offset + items.length`

Applies to: `items.list`, `items.getRecent`, `items.getTrash`, `collections.list`, `collections.getItems`, `notes.list`, `notes.search`, `attachments.list`, `search.quick`, `search.advanced`, `search.fulltext`, `search.byTag`, `search.byIdentifier`, `tags.list`.

**No bare arrays.** Methods that previously returned `[item, item, ...]` must wrap in the envelope.

### 3.2 Single entity

Methods returning one entity return the serialized object directly (no envelope):

```json
{"id": 42, "key": "YR5BUGHG", "title": "...", ...}
```

Applies to: `items.get`, `items.create`, `items.update`, `collections.get`, `collections.create`, `notes.get`, `notes.create`, `notes.update`.

### 3.3 Mutation confirmation

All destructive/state-change operations return:

```json
{"ok": true, "id": 42}
```

The `id` is always the numeric Zotero ID of the affected entity. For batch operations:

```json
{"ok": true, "ids": [1, 2, 3], "count": 3}
```

Applies to: `items.delete`, `items.trash`, `items.restore`, `items.batchTrash`, `collections.delete`, `collections.addItems`, `collections.removeItems`, `tags.add`, `tags.remove`, `tags.rename`, `tags.delete`, `notes.delete`, `attachments.delete`.

No `{deleted: true}`, `{trashed: true}`, `{added: true}`, `{removed: true}`, `{ok: true}` variety. One shape.

### 3.4 Domain-specific shapes (exceptions)

These methods return unique shapes justified by their domain:

| Method | Shape | Justification |
|--------|-------|---------------|
| `items.getFullText` | `{id, attachmentId, content, indexedChars, totalChars}` | Fulltext-specific metadata |
| `items.findDuplicates` | `{groups: number[][], totalGroups}` | Grouped item IDs |
| `collections.tree` | `[{...collection, children: [...]}]` | Recursive tree structure |
| `collections.stats` | `{id, name, items, attachments, notes, subcollections}` | Per-collection counts |
| `attachments.getPath` | `{id, path}` | Simple lookup |
| `annotations.list` | `[{id, type, text, comment, color, pageLabel, position, ...}]` | Annotation-specific fields |
| `export.bibliography` | `{format, html, text, style, count}` | Dual-output (HTML + plain) |
| `export.*` (others) | `{format, content, count}` | Export content |
| `rag.searchHits` | `{hits: [...], total, retrieval: {...}}` | RAG-specific metadata |
| `system.*` | Various | Introspection/meta, heterogeneous by design |

## 6. Error Strategy

### 6.1 Error shape

```json
{
  "code": -32602,
  "message": "Item 999 not found"
}
```

Codes:
- `-32600` — invalid JSON-RPC request
- `-32601` — method not found
- `-32602` — invalid params (caller's fault: missing item, bad input, wrong type)
- `-32603` — internal error (our fault or Zotero's fault)

### 6.2 Helpful error messages

Error messages should hint at correct usage:

| Bad | Good |
|-----|------|
| `"No arguments provided"` | `"Missing required parameter: parentId"` |
| `"Method not found: items.getNotes"` | `"Method not found: items.getNotes. Did you mean notes.list?"` |
| `"Not an attachment: 5925"` | `"Item 5925 is a journalArticle, not an attachment. Use items.getFullText to auto-find the PDF."` |

### 6.3 Method-not-found suggestions

When `processRequest` encounters an unknown method, fuzzy-match against registered methods and include a `"suggestion"` field:

```json
{
  "code": -32601,
  "message": "Method not found: items.getFullText",
  "data": {"suggestion": "items.getFullText"}
}
```

## 7. Introspection

### 7.1 `system.listMethods`

Returns `string[]` of all registered method names, sorted alphabetically.

```json
["annotations.create", "annotations.delete", "annotations.list", "attachments.add", ...]
```

### 7.2 `system.describe`

Without params: returns all methods with their parameter schemas.
With `{method: "items.get"}`: returns schema for one method.

```json
{
  "name": "items.get",
  "description": "Fetch a single item's full metadata by ID or key",
  "params": {
    "id": {"type": "number | string", "required": true, "description": "Numeric ID or 8-char item key"}
  },
  "returns": "Single serialized item object"
}
```

Implementation: store metadata alongside each `registerHandlers` call. The schema object is the source of truth — keep it close to the handler code, not in a separate file.

## 8. CLI Layer Specification

### 8.1 Architecture

```
Agent (Claude Code)
  ↓ bash tool (245 tokens, trained-in)
zotron CLI (typer, noun-verb, --json)
  ↓ HTTP POST localhost:23119/zotron/rpc
XPI JSON-RPC handlers (Zotero process)
```

The CLI is a thin typed wrapper over `ZoteroRPC.call()`. It adds:
- Argument validation + helpful error messages before the RPC call
- `--json` structured output (default for agents) vs human-readable (default for terminals)
- `--jq` post-processing filter
- Exit codes with semantic meaning
- `--help` on-demand discovery

### 8.2 CLI design rules

**Noun-verb structure** (matches gh, kubectl):
```bash
zotron items get 12345
zotron items fulltext 12345
zotron search quick "数字经济" --limit 10
zotron collections list
zotron notes list --parent 12345
zotron tags add 12345 --tag "核心期刊"
```

**`--json` flag on every command:**
```bash
# Human-readable (default in terminal)
zotron search quick "数字经济"

# Structured JSON (default when stdout is piped / used by agent)
zotron search quick "数字经济" --json
```

Auto-detect: if stdout is not a TTY (piped or captured by agent), default to JSON output. Agents never need to remember `--json`.

**Flags match RPC parameter names:**
```bash
# CLI flag                    → RPC parameter
--limit 10                    → "limit": 10
--parent 12345                → "parentId": 12345
--tag "核心期刊"               → "tag": "核心期刊"
```

Exception: positional arguments for the most common parameter (usually `id` or `query`).

**Meaningful exit codes:**
| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Generic error |
| 2 | Usage error (bad flags, missing args) |
| 3 | Not found (item, collection, method) |
| 4 | Zotero unavailable |

**Error output to stderr, data to stdout.** Agents can safely capture stdout as JSON without error messages mixed in.

### 8.4 Agent-optimized output

**List output** auto-formats for agents:

```bash
$ zotron search quick "数字经济" --limit 3
# stdout (JSON when piped):
{
  "items": [
    {"id": 356, "key": "IGFDXZKP", "title": "工业机器人应用...", ...},
    ...
  ],
  "total": 42,
  "limit": 3,
  "offset": 0,
  "hasMore": true
}
```

**Fulltext output** returns content directly:

```bash
$ zotron items fulltext 12345
# stdout: raw text content (efficient, no JSON envelope overhead)
# stderr: metadata (id, attachmentId, indexedChars, totalChars)
```

When `--json` is explicit, wrap in envelope.

## 9. Skill Layer Responsibilities

The skill docs (`claude-plugin/skills/zotero/`) teach agents workflows, not API mechanics:

1. **Workflow routing** — "want to read a paper? `zotron items fulltext 12345`. Want OCR text? `zotron notes list --parent 12345` then filter by `ocr` tag. Want RAG spans? `zotron rag search`."
2. **Common recipes** — multi-step operations like "OCR → index → search"
3. **CLI examples** — every workflow step shows the CLI command, not raw RPC

The skill docs should NOT:
- Duplicate parameter schemas (agents use `--help` or `zotron system describe`)
- List all methods (agents use `zotron system list-methods`)
- Show raw RPC calls (agents use CLI subcommands)

## 10. Migration Plan

### Phase 1: CLI parity (highest priority)

Close the coverage gap — bring CLI from 32 to 85+ subcommands:
- Add all missing `items` subcommands: `list`, `create`, `update`, `delete`, `fulltext`, `recent`, `related`, `citation-key`
- Add all `notes` subcommands: `list`, `get`, `create`, `update`, `delete`, `search`

- Add `annotations` namespace: `list`, `create`, `delete`
- Add `settings` subcommands: `get`, `set`, `list`
- Add `system` subcommands: `list-methods`, `describe`
- Add `--json` flag globally (auto-detect TTY for default)
- Add semantic exit codes (0/1/2/3/4)

### Phase 2: RPC normalization

Fix the RPC substrate that CLI calls:
- Rename `notes.get` → `notes.list`, add real `notes.get`
- Rename `getFulltext` → `getFullText`
- Extend `requireCollection` to accept key/name
- Normalize parameter names (`itemId` → `id`, `relatedId` → `targetId`)
- Wrap all bare-array returns in list envelope
- Normalize mutation returns to `{ok, id}`
- Add `system.listMethods`, `system.describe`

### Phase 3: Skill docs rewrite

Update all skill docs to use CLI commands instead of raw RPC:
- `search.md`: all examples as `zotron search quick ...` not `zotron rpc search.quick '{...}'`
- `rag.md`: use `zotron rag search ...`
- `ocr.md`: add "reading OCR results" section with `zotron notes list --parent ...`
- `SKILL.md`: update workflow table
- Remove `rpc` examples from primary docs (keep only in escape-hatch reference)

### Phase 4: Cleanup

- Remove `items.getAttachments`, `items.getNotes` (convenience methods from v1)
- Remove deprecated annotation aliases from `notes.*`
- Remove `rag.searchCards` if no consumers
- Bump minor version

Each phase is one commit on dev. Phase 1 is the highest priority — it directly fixes the agent experience.
